## Groups: The Foundation

Groups are the basic organizational unit. Every page belongs to a group.

### Basic Group Configuration

```typescript
{
  id: "getting-started",
  label: "Getting Started",
  icon: "ğŸš€",
  entries: [
    { slug: "getting-started/introduction" },
    { slug: "getting-started/installation" },
    { slug: "getting-started/quick-start" },

```

A group needs:

- **id**: Unique identifier (usually matches folder name)
- **label**: Display name in the sidebar
- **icon**: Emoji or SVG icon reference
- **entries**: List of pages (or use `autoGenerated: true`)

### Auto-Generated Groups

Let the system discover pages automatically:

```typescript
{
  id: "features",
  label: "Features",
  icon: "âœ¨",
  autoGenerated: true,

```

This automatically includes all files from `content/docs/features/` in alphabetical order.

## Tabs: Major Sections

Tabs create distinct sections in your documentation. They appear as clickable tabs in the sidebar, and only one tab's content is visible at a time.

### Creating a Tab

To make a group into a tab, add `tab: true`:

```typescript
{
  id: "patterns",
  label: "Common Patterns",
  icon: "ğŸ¯",
  tab: true,
  autoGenerated: true,
}
```

This creates a new tab called "Common Patterns" that users can click to see a different set of documentation.

### Default Tab

Every collection has a default tab that contains groups without `tab: true`. Configure it like this:

```typescript
export const SIDEBAR_NAVIGATION: SidebarNavigation = {
  docs: {
    defaultTab: {
      label: "Documentation",
      icon: "document",
    },
    groups: [
      // Groups without tab: true appear in the default tab
      { id: "getting-started", label: "Getting Started", icon: "ğŸš€" },
      { id: "features", label: "Features", icon: "âœ¨" },

      // Groups with tab: true create separate tabs
      { id: "patterns", label: "Patterns", icon: "ğŸ¯", tab: true },
    ],
  },
};
```

## When to Use Tabs

Tabs are perfect for:

### Separating Major Concerns

```typescript
groups: [
  // Default tab: Main documentation
  { id: "getting-started", label: "Getting Started", icon: "ğŸš€" },
  { id: "features", label: "Features", icon: "âœ¨" },
  { id: "configuration", label: "Configuration", icon: "âš™ï¸" },

  // Separate tab: Examples and patterns
  { id: "patterns", label: "Common Patterns", icon: "ğŸ¯", tab: true },

  // Separate tab: Help resources
  { id: "help", label: "Help & Support", icon: "â“", tab: true },
]
```

### Different Content Types

- **Main docs** in default tab
- **API reference** in a separate tab
- **Tutorials** in another tab
- **FAQ/Support** in a help tab

### Reducing Sidebar Clutter

If your default tab is getting too long, move some groups to tabs:

```typescript
// Before: Long sidebar
groups: [
  { id: "intro" },
  { id: "guides" },
  { id: "reference" },
  { id: "examples" },
  { id: "patterns" },
  { id: "recipes" },
  { id: "faq" },
  { id: "troubleshooting" },
]

// After: Organized with tabs
groups: [
  // Default tab
  { id: "intro" },
  { id: "guides" },
  { id: "reference" },

  // Examples tab
  { id: "examples", tab: true },
  { id: "patterns", tab: true },
  { id: "recipes", tab: true },

  // Help tab
  { id: "faq", tab: true },
  { id: "troubleshooting", tab: true },
]
```

## When NOT to Use Tabs

Avoid tabs when:

- **You have few pages**: 1-2 groups don't need tabs
- **Content is sequential**: Tutorials that build on each other should stay together
- **Users need to reference across sections**: If users constantly switch between sections, keep them in one tab

## Tab Behavior

### Navigation State

When users switch tabs:

- The sidebar updates to show only that tab's groups
- The URL doesn't change (tabs are UI-only)
- The current page remains visible if it's in the new tab

### Tab Persistence

The selected tab persists across page navigation:

- If you're viewing the "Patterns" tab and click a link, you stay in that tab
- Switching tabs is an explicit user action

## Nested Groups

You can nest groups within tabs for deeper organization:

```typescript
{
  id: "api",
  label: "API Reference",
  icon: "ğŸ“š",
  tab: true,
  entries: [
    // You can manually list entries from different folders
    { slug: "api/authentication/overview" },
    { slug: "api/endpoints/users" },
    { slug: "api/endpoints/posts" },
  ],
}
```

However, keep nesting shallow (max 3 levels: tab > group > page) for better usability.

## Best Practices

### Keep Tabs Focused

Each tab should have a clear purpose:

- âœ… "Documentation" vs. "API Reference" vs. "Examples"
- âŒ "Stuff" vs. "More Stuff" vs. "Other Things"

### Limit Tab Count

Too many tabs is overwhelming:

- âœ… 2-4 tabs is ideal
- âš ï¸ 5-6 tabs is manageable
- âŒ 7+ tabs is too many

### Use Descriptive Labels

Make it obvious what's in each tab:

- âœ… "Getting Started", "API Reference", "Examples"
- âŒ "Docs", "Reference", "Other"

### Group Related Content

Keep related pages together in the same group:

- âœ… All authentication docs in one group
- âŒ Authentication scattered across multiple groups

### Consider User Journey

Organize tabs by how users will navigate:

- **Learning path**: Getting Started â†’ Guides â†’ Reference
- **Task-based**: By feature or use case
- **Role-based**: User docs vs. developer docs
